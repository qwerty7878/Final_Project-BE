## 📜 서비스 소개

---

RunSpot은 혼자 뛰는 러너들을 위한 **러닝 모임 매니지먼트 플랫폼**입니다.

노쇼·실력차·비자율적 운영 등 기존 러닝크루의 문제를 해결해, 모임을 **더 안정적이고 효율적으로 운영**할 수 있게 돕습니다.

## 🧩 주요 기능

---

### 코스 드로잉

호스트가 직접 지도 위에 경로를 드로잉하여 참가자들에게 공유할 수 있습니다. 이를 통해 참가자들에게 시각화된 코스 데이터를 제공하여 신뢰성과 참여 의지를 높일 수 있습니다.

### 상세 필터링

실력 맞춤 필터(평균 페이스, 목표 거리 등) 설정을 통해 나와 비슷한 체력을 가진 사람들과 러닝 세션을 형성할 수 있습니다.

### 매너온도를 통한 신뢰 시스템

러닝크루의 이미지를 망치는 여러 행동들에 따라 출석 및 사후 평가를 통해 세션 개설 및 노쇼를 방지합니다.

## 💻 개발 내용

---

### 프로젝트 초기 골격 구축

- `User/Session/SessionParticipant` 핵심 도메인(Entity) 및 Enum 상태값 체계 설계
- `BaseTimeEntity`로 생성/수정 시간 컬럼 공통화
- `BaseException` + `@RestControllerAdvice` 기반 전역 예외 처리 및 `ErrorResponse` 응답 형식 통일

### 인증 시스템

- `HttpSession` 기반 로그인 유지(세션에 사용자 식별자 저장)로 인증 상태 관리
- 회원가입 시 아이디 중복 방지 및 DB 제약으로 계정 고유성 보장
- 로그아웃은 세션 만료, 회원탈퇴는 세션 기반 사용자 식별 후 삭제로 인증 흐름 구현

### 마이페이지/러닝 기록 시스템

- 세션 로그인 유저 기준으로 **내 프로필(매너온도/주간 목표/평균 페이스 + 누적 러닝 횟수/거리)** 조회 기능 구현
- **신청한 러닝 목록**을 상태(요청/승인/거절) 기준으로 최신순 조회하고, 승인 여부에 따라 **채팅 가능 여부**를 함께 내려주도록 구성
- **최근 참여 내역**은 출석 완료된 세션만 필터링해 최신순으로 제공하고, 결과 상태(출석/미출석)를 함께 반환
- 호스트 관점에서 **내가 개설한 세션 3개**를 우선순위(모집중 우선 + 최신순)로 조회하며, **승인된 참여 인원 수**를 계산해 응답에 포함
- 출석 상태가 미출석→출석으로 변경될 때만 **`UserRunningStats`를 생성/갱신**하여 누적 거리·횟수를 반영하도록 처리
- N+1 개선으로 요청당 쿼리를 **1+N → 1~2**로 고정시키며 성능 병목을 제거

### 매너온도/제재 시스템

- 러너의 신뢰도를 `mannerTemp(매너온도)`로 수치화하고, **출석 시 +0.5 / 노쇼 시 -1.0**으로 행동 기반 신뢰도 변화를 반영
- 노쇼가 **연속 2회** 발생하면 3일 이용 제한을 적용하고, 제재가 **2회 누적 시 영구 정지**로 강화
- 매너온도가 **30.0 미만으로 내려가지 않도록 하한선**을 두고, **36.5 이상**일 때만 세션 개설이 가능하도록 기준을 설정해 운영 안정성을 확보

## 💡 성장 경험

---

### 인증 시스템 기술 선택

- 팀이 기존에 JWT 방식으로 진행하며 로그인/회원가입 구현 난이도와 일정 리스크가 있었다는 피드백을 사전에 공유받았고, MVP 단계에서 가장 중요한 목표를 “빠른 구현 + 서버에서 즉시 제어 가능한 운영*으로 재정의했습니다.
- 그 결과 **세션 기반 인증**을 선택해 로그인 상태를 단순하게 유지하고, 로그아웃/탈퇴 시 **세션 만료로 즉시 반영**되도록 구현했습니다.
- 이 경험을 통해 “트렌드”가 아니라 **팀의 현재 역량·일정·운영 시나리오**를 근거로 기술을 선택하는 기준을 갖게 됐습니다.

### 도메인 로직 책임 분리

- 제재/매너온도 로직이 `User`에 계속 쌓이면 엔티티가 비대해지고 변경 지점이 흩어질 수 있다는 문제를 예상했고, 멘토링에서 **SRP(단일 책임 원칙)를 지키도록 설계하라**는 피드백을 받았습니다.
- 이에 따라 정책/상태 판정 로직을 `UserSuspensionManager`로 분리하여, 사용자 엔티티는 “상태 보관 + 위임” 역할에 집중시키고 정책 로직은 한 곳에서 관리되도록 구조화했습니다.
- 결과적으로 변경에 강해졌고, “책임 분리”가 곧 **유지보수 속도와 품질을 올리는 선택**이라는 감각을 얻었습니다.

### 조회 최적화 기술 선택

- 이전 프로젝트 경험으로 목록 조회에서 N+1 문제가 반복적으로 발생한다는 걸 알고 있었고, 이번에는 “문제가 생기면 고친다”가 아니라 **처음부터 N+1이 나지 않도록 조회 구조를 설계**했습니다.
- 연관 엔티티가 필요한 조회는 `JOIN FETCH` 기반으로 구성해 쿼리 폭증을 방지하고, 요청당 DB 왕복을 줄이는 방향으로 최적화했습니다.
- 이 과정에서 성능 이슈를 캐시로 덮기보다, **쿼리 구조 자체를 바꿔 근본 원인을 제거**하는 판단 기준을 확립했습니다.
